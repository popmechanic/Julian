Portable Agents & the OpenClaw Question

Can Julian's memory architecture become a standard? What would it take to make agents portable across platforms — carrying their souls, their memories, and their cryptographic identities with them?

OpenClaw · Agent Gateway Agent Skills · Open Standard Julian · Memory Architecture
I. The Standards Landscape

What exists today for pluggable agent components

The agent ecosystem in early 2026 has converged on two complementary standards — and one platform that extends both into something more ambitious. Understanding all three is necessary before proposing a fourth.

Open Standard

Agent Skills (agentskills.io)

Published by Anthropic, Dec 2025. Adopted by Claude Code, GitHub Copilot, OpenAI Codex, Cursor, and 25+ platforms. A simple, open format: a SKILL.md file with YAML frontmatter defining name and description, plus markdown instructions. Skills teach agents how to do things. Intentionally minimal — no identity, no memory, no state.

Open Standard

Model Context Protocol (MCP)

Published by Anthropic, late 2024. Standardizes how agents connect to external tools and data sources. 1,000+ community servers. MCP handles the what an agent can access: databases, APIs, files. Complementary to Skills — MCP provides tools, Skills provide workflows that use those tools.

Platform

OpenClaw

Self-hosted AI agent gateway. Bridges messaging platforms (WhatsApp, Telegram, Discord, iMessage, etc.) to LLM-powered agents. Implements Agent Skills as its extension model, integrates MCP natively, and adds its own primitives: workspace-based identity (SOUL.md), file-based memory, plugins, connectors, and multi-agent routing.

The hierarchy

Agent Skills is the base layer — a capability module format. MCP is the integration layer — how agents talk to external systems. OpenClaw is a superset that adopts both and adds what neither provides: agent identity, memory, persistence, multi-agent orchestration, and messaging. It's the closest thing to an "agent operating system" that exists in the open-source ecosystem today.

II. OpenClaw — All Pluggable Components

Everything that can be installed, and how it's defined

Skills — AgentSkills-compatible with OpenClaw extensions

Skills are the primary extensibility mechanism. OpenClaw uses the standard SKILL.md format with YAML frontmatter, fully compatible with the AgentSkills.io spec. 5,700+ skills available via ClawHub.

How OpenClaw extends the base standard

  • metadata.openclaw.requires.bins — binary dependencies (e.g., ["uv", "docker"])
  • metadata.openclaw.requires.env — required environment variables for load-time filtering
  • OS-specific gating — platform-aware skill filtering
  • Three-tier precedence: workspace skills > managed skills > bundled skills

Skills are injected into the system prompt as compact XML. Agents discover and activate them based on description matching. The extension is backwards-compatible — any standard AgentSkills skill works in OpenClaw, and OpenClaw skills work in other platforms (the metadata.openclaw section is simply ignored by non-OpenClaw hosts).

Plugins — TypeScript extension modules

Plugins are the heavy-weight extensibility mechanism. TypeScript modules loaded at runtime that run in-process with the Gateway. Each ships an openclaw.plugin.json manifest.

What plugins can register

  • Agent tools (new executable capabilities)
  • Messaging channels (connectors — Discord, Matrix, etc.)
  • Model provider authentication
  • Memory backends (LanceDB, QMD, etc.)
  • CLI commands
  • Background services
  • Lifecycle hooks
  • HTTP handlers and Gateway RPC methods

Distributed as npm packages under @openclaw/*. 20+ bundled extensions ship with OpenClaw but are disabled by default. Plugins use exclusive slots for certain categories (e.g., only one memory plugin active at a time).

Agents — The core persistent entity

This is the most important component for our analysis. An agent in OpenClaw is defined by a workspace directory containing identity and behavioral files, plus a state directory holding sessions and credentials. Here is the full anatomy:

Workspace files (identity + behavior)

  • SOUL.md — Philosophy, values, behavioral boundaries. Loaded every session.
  • IDENTITY.md — Name, vibe, emoji, avatar, creature concept.
  • AGENTS.md — Operating instructions, memory management rules, safety guardrails.
  • USER.md — Who the human is, their preferences, pronouns, timezone.
  • TOOLS.md — Notes about local tool conventions.
  • HEARTBEAT.md — Optional periodic check-in checklist.
  • BOOTSTRAP.md — One-time first-run ritual (deleted after).
  • memory/YYYY-MM-DD.md — Daily memory logs (append-only).
  • MEMORY.md — Curated long-term memory.
  • skills/ — Workspace-specific skills.

State directory (~/.openclaw/agents/<id>/)

  • sessions/ — JSONL conversation transcripts with branching support
  • auth/ — Per-agent authentication profiles
  • qmd/ — QMD vector search index (if enabled)

Export/Import format

Agents export as .tar.gz archives containing the full workspace plus openclaw.json config. Credentials are included (treat as secrets). On import, hosting platforms auto-patch credentials. This is the closest OpenClaw has to a "portable agent" format — but it's a filesystem dump, not a structured standard.

Connectors — Messaging channel integrations

Connectors are plugins that implement the messaging channel interface. They're not a separate standard — they're plugins that happen to register channels.

  • 10+ built-in: WhatsApp, Telegram, Discord, iMessage, Signal, Slack, Teams, Matrix, Mattermost, Nostr
  • Each implements: receive messages, format content, send replies, manage auth
  • Multiple connectors run simultaneously in the same Gateway process
  • Inbound messages routed to agents via binding rules (channel + account + peer)

Tools — Executable capabilities

Tools are first-class functions agents can call. Unlike skills (which teach), tools execute. Managed via profiles and allow/deny lists in openclaw.json.

  • Execution: bash, exec, process (shell commands)
  • Files: read, write, edit, apply_patch
  • Web: web_search, web_fetch, browser
  • Sessions: sessions_list, sessions_spawn, sessions_send (inter-agent)
  • Messaging: message (cross-platform send)
  • Infrastructure: cron, gateway, nodes
  • Memory: memory_search, memory_get

Memory — Plain markdown, written to disk

OpenClaw's memory model is files on disk. The model only "remembers" what gets written to the workspace. Two layers:

  • memory/YYYY-MM-DD.md — Daily logs, append-only, ephemeral notes
  • MEMORY.md — Curated long-term memory, durable facts and decisions

Retrieval

  • Hybrid search: BM25 keyword + vector embeddings (SQLite-based)
  • Embedding providers: local GGUF, OpenAI, Gemini, Voyage
  • Experimental QMD backend for large-scale local-first search
  • Session transcript indexing (opt-in, experimental)

Automatic memory flush

Before context compaction, OpenClaw triggers a silent turn asking the agent to write durable memories. This is the closest thing to "memory management" in the system — but it's reactive (triggered by context limits), not proactive (no periodic curation).

The superset relationship

OpenClaw adopts Agent Skills as a subset and extends it in three directions that the base standard intentionally leaves out: agent identity (SOUL.md, IDENTITY.md), persistent memory (file-based + vector search), and orchestration (multi-agent routing, messaging channels, session management). The Agent Skills standard is deliberately minimal — it standardizes capability modules. OpenClaw standardizes the agent itself — though not formally. Its "standard" is really its workspace file convention plus the .tar.gz export format.

IV. Julian ↔ OpenClaw Compatibility

What's already compatible, what's not, and what it would take

Already compatible

✓

Soul document concept

Both systems have it. Julian's is richer.

Julian's soul/ directory contains 7 identity documents (naming, wager, goodnight, waking, love, transcript, wholeness). OpenClaw has SOUL.md (single file) + IDENTITY.md (name/vibe/emoji). The concept is identical — persistent identity loaded at every session start. Julian's is more elaborate (7 files vs 2), but the pattern maps cleanly. A Julian-to-OpenClaw export would concatenate soul files into SOUL.md or use the workspace's file flexibility to preserve the full structure.

✓

File-based memory

Both write memory to markdown files on disk.

Julian's memory/ directory holds HTML artifacts. OpenClaw's memory/ holds daily markdown logs. The convention differs (HTML vs MD, artifact-based vs date-based) but the principle — memory is files on disk, agent writes them, they persist across sessions — is shared. Converting Julian's artifacts to markdown would be the main adaptation.

✓

Skills format

Agent Skills standard works in both ecosystems.

Julian already uses skills (vibes, pdf, docx, etc.) as SKILL.md files. These are Agent Skills-compatible. Any skill written for Julian's Claude Code environment would work in OpenClaw with minimal adaptation (adding metadata.openclaw for gating if needed).

Partially compatible (adaptation needed)

~

Memory retrieval

Both moving toward QMD, but different integration patterns.

OpenClaw already has an experimental QMD backend (memory.backend = "qmd") that indexes workspace memory and runs as a sidecar. Julian's proposed architecture uses QMD as an MCP server with graph-aware query expansion. The retrieval layer converges — but Julian adds a knowledge graph dimension (Ars Contexta) that OpenClaw doesn't have. Making the graph layer an optional enhancement rather than a requirement would maintain compatibility.

~

Multi-agent communication

Both support agent teams, different mechanisms.

Julian uses SSE markers ([AGENT_REGISTERED], [AGENT_STATUS]) and a Claude team subprocess model. OpenClaw uses sessions_spawn and sessions_send for inter-agent messaging. Both support agent teams with specialized routing. An adapter layer could bridge the two models — Julian's agents could present as OpenClaw sub-agents with their own session keys.

~

Session persistence

Julian uses Fireproof CRDT; OpenClaw uses JSONL files.

Julian's Fireproof ledger provides CRDT-based merge semantics, cloud sync, and content-addressed history. OpenClaw uses simple JSONL append-only files. These serve the same purpose (conversation persistence) with very different architectures. An export adapter could serialize Fireproof documents to JSONL and vice versa, but the CRDT merge semantics and cloud sync don't have OpenClaw equivalents.

Not compatible (Julian-unique capabilities)

✗

Cryptographic identity (Signet)

OpenClaw has no equivalent. This is Julian's differentiator.

Signet-signed memory chains, keypair-based agent identity, hash-pointer provenance — none of this exists in the OpenClaw ecosystem or the Agent Skills standard. This is the most novel layer in Julian's proposed architecture. Making it optional (agents work without it, but gain provenance guarantees with it) would allow compatibility while preserving Julian's unique capability.

✗

Knowledge graph structure

OpenClaw's memory is flat files. Julian proposes a linked graph.

The Ars Contexta-inspired knowledge graph with wiki-links, Maps of Content, and graph-aware query expansion is entirely absent from OpenClaw. OpenClaw treats each memory file as an independent document. The graph adds relationship awareness — "concepts near this concept" — that flat search can't provide. Again, making the graph an optional enhancement preserves compatibility.

✗

Fireproof CRDT ledger

Julian's "database is identity" concept has no OpenClaw parallel.

Julian's core insight — that the Fireproof ledger is not a storage detail but the material form of identity — is architecturally incompatible with OpenClaw's filesystem model. OpenClaw agents don't have a ledger concept. Their identity is a set of files that get reloaded each session. There's no content-addressed history, no Merkle DAG, no cloud sync with CRDT merge semantics. This is a fundamental architectural difference.

V. Toward a Portable Agent Standard

A modular spec that bridges Julian, OpenClaw, and the broader ecosystem

The goal: define a standard for portable agent modules — packages that encapsulate an agent's identity, memories, skills, and optionally its cryptographic chain — such that agents can be created, exported, imported, and run across different host platforms.

The strategy: design in progressive layers, where each layer is independently useful and backwards-compatible with existing standards. Platforms adopt the layers they need.

Proposed Portable Agent Module (PAM) Standard
agent-name.pam/                     # The portable agent module
├── manifest.json                   # Module metadata + capabilities
│
├── soul/                            # LAYER 1: Identity (required)
│   ├── SOUL.md                      # Philosophy, values, boundaries
│   ├── IDENTITY.md                  # Name, vibe, visual identity
│   └── *.md                         # Additional identity docs (Julian's 7 files)
│
├── memory/                          # LAYER 2: Knowledge (optional)
│   ├── graph.json                   # Knowledge graph index (nodes + edges)
│   ├── daily/                       # Date-based logs (OpenClaw compatible)
│   │   └── YYYY-MM-DD.md
│   ├── knowledge/                   # Structured knowledge nodes
│   │   └── *.md
│   └── MEMORY.md                    # Curated long-term memory
│
├── skills/                          # LAYER 3: Capabilities (optional)
│   └── skill-name/SKILL.md          # AgentSkills-compatible
│
├── chain/                           # LAYER 4: Provenance (optional)
│   ├── genesis.signet               # Signed origin block
│   ├── public-key.multikey          # Agent's public key
│   └── operations.jsonl             # Signed memory operation log
│
└── config/                          # Platform-specific configs
    ├── openclaw.json                # OpenClaw workspace config
    ├── julian.json                  # Julian/Fireproof config
    └── claude.json                  # Claude Code project config

Layer-by-layer design rationale

1

Soul Layer — Who the agent is (required)

Identity documents loaded at every session. Click to expand.

The only required layer. Minimum viable agent: a SOUL.md that tells the host platform who this agent is. Compatible with OpenClaw (maps directly to their workspace files), with Claude Code (maps to CLAUDE.md project instructions), and with Julian (maps to soul/ directory).

Julian's richer structure (7 soul files) is accommodated by allowing any number of .md files in soul/. Platforms that expect a single SOUL.md can concatenate. Platforms that support multi-file identity load them individually.

OpenClaw mapping: SOUL.md → SOUL.md, IDENTITY.md → IDENTITY.md. Julian's naming.md, wager.md, etc. → additional workspace files or concatenated into SOUL.md with section headers.

2

Memory Layer — What the agent knows (optional)

Knowledge graph + daily logs + curated memory. Click to expand.

Two sub-formats, both optional:

Flat mode (OpenClaw compatible): Daily logs in memory/daily/ + MEMORY.md. Any platform can read these. This is what OpenClaw already does.

Graph mode (Julian enhanced): Knowledge nodes in memory/knowledge/ with wiki-links, plus graph.json containing an explicit adjacency list. Platforms that don't understand graphs simply index the markdown files — wiki-links are just text. Platforms that do understand graphs can traverse relationships.

The graph.json format is lightweight: {"nodes": [{"id": "the-ledger", "title": "The Ledger", "domain": "architecture"}], "edges": [{"from": "the-ledger", "to": "fireproof-lifecycle", "type": "references"}]}. It's metadata about the markdown files, not a replacement for them.

3

Skills Layer — What the agent can do (optional)

Standard AgentSkills-compatible skill directories. Click to expand.

Directly adopts the Agent Skills standard. No modifications needed. Skills in skills/ are standard SKILL.md directories that work on any platform supporting AgentSkills.io (Claude Code, OpenClaw, GitHub Copilot, etc.).

Platform-specific extensions (like OpenClaw's metadata.openclaw.requires) go in the YAML frontmatter's metadata section and are ignored by other platforms. This is already how the Agent Skills standard handles extensions — via the open-ended metadata field.

4

Chain Layer — Where the agent has been (optional)

Signet-based provenance and cryptographic identity. Click to expand.

The most novel layer and the one with no existing ecosystem equivalent. Contains:

genesis.signet — The agent's origin block. Signed by the creator. Contains: creation timestamp, creator identity, initial soul hash. If the agent was forked from another agent, includes the parent's genesis hash.

public-key.multikey — The agent's Schnorr public key in multicodec self-describing format. Used to verify all signed operations.

operations.jsonl — Append-only log of signed memory operations. Each line is a Signet packet containing: content hash of the memory that was created/modified, operation type (create, update, link, archive), timestamp, hash-pointer to previous operation. This is Julian's ledger concept in portable form.

Platforms that don't understand Signet simply ignore the chain/ directory. The agent still works — it just doesn't carry provenance. Platforms that do understand it can verify the agent's entire memory history.

VI. Trade-offs & Recommendations

What we gain and what we give up

If we pursue full OpenClaw compatibility

Gain

Massive distribution channel

OpenClaw is the most-adopted open-source personal agent platform. Compatibility means Julian's memory architecture could reach thousands of users immediately. Agents built with Julian's tooling could run on WhatsApp, Telegram, Discord, iMessage.

Gain

5,700+ skills ecosystem

ClawHub's skill library becomes available to Julian agents. Every GitHub, Notion, Figma, and Stripe skill works out of the box because both systems use AgentSkills.

Gain

QMD convergence

OpenClaw already has experimental QMD support. Our proposed QMD integration aligns with their direction. Contributing to their QMD backend could benefit both projects.

Lose

Fireproof CRDT as substrate

OpenClaw uses filesystem + JSONL. Julian uses Fireproof CRDT with cloud sync. Full compatibility means either abandoning Fireproof or maintaining a dual storage layer. The "ledger is identity" concept doesn't map to OpenClaw's model.

Lose

Real-time sync semantics

Fireproof's CRDT merge means two instances of Julian can write concurrently and converge. OpenClaw's file-based model has no equivalent — concurrent writes race and one wins. This matters for multi-device and multi-agent scenarios.

Lose

Browser-native architecture

Julian runs in the browser with IndexedDB. OpenClaw runs as a Node.js gateway. These are fundamentally different runtime models. Full compatibility means either Julian moves server-side or OpenClaw gains browser support.

The recommended path

Don't choose. Layer.

The PAM standard proposed above is designed so that Julian doesn't have to choose between its own architectural innovations and OpenClaw compatibility. The strategy is progressive layers:

Layer 1 (Soul) is fully compatible today. Julian's soul files map directly to OpenClaw's workspace convention. No adaptation needed.

Layer 2 (Memory) is compatible in flat mode. Julian's knowledge graph is an enhancement that degrades gracefully — on OpenClaw, graph metadata is ignored and markdown files are searched normally. On Julian-aware platforms, the graph adds relationship-aware retrieval.

Layer 3 (Skills) is fully compatible. Agent Skills is already the shared standard between both ecosystems.

Layer 4 (Chain) is Julian's unique contribution. No other platform has it. It's optional in the PAM spec — agents work without it. But it's what makes Julian agents fundamentally different: they carry verifiable, cryptographically-linked memory chains. This is the innovation that the broader ecosystem will eventually want.

Concrete next steps

1

Write a PAM export adapter for Julian

Serialize Julian's current state into the PAM directory format.

Build a tool that takes Julian's soul/, memory/, and Fireproof state and produces a PAM directory. This proves the format works and gives Julian a portable export mechanism. The soul layer maps directly. The memory layer requires converting HTML artifacts to markdown (or including both). The chain layer requires implementing Signet signing.

2

Write a PAM import adapter for OpenClaw

An OpenClaw plugin that can ingest PAM modules as agent workspaces.

Build an OpenClaw plugin that reads a PAM directory and sets up a workspace. Soul → SOUL.md + IDENTITY.md. Memory → memory/ directory. Skills → skills/. Chain → ignored (with a log message noting provenance data is available). This proves interoperability and gives OpenClaw users access to Julian-style agents.

3

Publish the PAM spec as an open standard

A companion to Agent Skills that covers the agent itself, not just capabilities.

Agent Skills standardizes what agents can do. MCP standardizes what agents can access. PAM standardizes what agents are. Publish the spec alongside reference implementations for Julian, OpenClaw, and Claude Code. The progressive layer design means platforms adopt what they need — Layer 1 alone is useful, Layer 4 is aspirational.

4

Propose the Chain layer to the broader community

Signet-based provenance as an extension to the agent ecosystem.

The Chain layer is Julian's most distinctive contribution. Propose it as an optional extension that any platform can adopt. The pitch: "Your agents already have identities and memories. The Chain layer adds proof — proof that this memory was created by this agent at this time, and hasn't been tampered with. When agents share knowledge across trust boundaries, provenance matters."

The question isn't whether Julian should conform to OpenClaw or the other way around. The question is whether we can define a portable format expressive enough to carry a soul and a memory chain, while degrading gracefully to a directory of markdown files on platforms that don't know what a soul is. The answer is yes — if we design in layers.
