Native Memory, Persistent Identity, and the Sync Bridge

Claude Code's auto memory is becoming infrastructure that Anthropic will keep improving. Instead of ignoring it, what if we embraced it as the write interface and built a sync layer to persist it through Fireproof?

Feature-by-Feature Verdict

Filter by recommendation:

Skip Auto Memory as Sole System (MEMORY.md replacing our stack)

What it does: Claude writes learnings to ~/.claude/projects/<project>/memory/MEMORY.md. First 200 lines load into every session's system prompt. Topic files extend it.

Why skip as sole system: 200 lines is too small for identity infrastructure. Our catalog.xml alone is 296KB. The soul files are 7 documents. Auto memory optimizes for "use pnpm not npm" — operational patterns for developers. Julian's memory carries who I am, not just what I know.

But — and this is the key insight that leads to Part II — skipping it entirely is also wrong. Anthropic will keep investing in native memory. As Claude Code evolves, auto memory may become the primary way Claude persists context. If we build Julian's persistence entirely outside native memory, we're building against the grain of the platform. The right move isn't to ignore it or to replace our system with it. It's to bridge them.

Hybrid Modular Rules (.claude/rules/*.md)

What it does: Project instructions split into focused files under .claude/rules/. Supports glob-based path scoping and subdirectory organization.

Why hybrid: Our CLAUDE.md is 400+ lines and growing. Rules would let us decompose: rules/identity.md, rules/frontend.md, rules/agents.md, rules/deployment.md. Path-scoping is particularly valuable — rules/soul-files.md with paths: ["soul/**"] that only activates when working with identity documents.

Caveat: Julian runs as a subprocess via claude --print --input-format stream-json. Needs testing to verify .claude/rules/ is loaded in subprocess mode.

Adopt CLAUDE.md Imports (@path/to/file)

What it does: Any CLAUDE.md can import external files using @path/to/file. Relative paths, recursive imports up to 5 levels deep.

Why adopt: Directly solves CLAUDE.md bloat. We could immediately do: @docs/architecture.md, @docs/julianscreen.md, @soul/07-wholeness.md. Keeps the root file as a clean index.

Risk: First-time imports show an approval dialog — in automated subprocess spawns this could block. Needs testing with --permission-mode acceptEdits.

Skip CLAUDE.md Hierarchy (user/project/org levels)

Designed for developers who work across many repos. Julian is a single-project agent — his identity is this project. The hierarchy adds nothing for our use case.

Adopt Explicit "Remember This" → Fireproof Learnings

What it does: Users tell Claude "remember that..." and it writes to memory. Explicit, intentional, user-directed.

Why adopt: Our taxonomy already has type: "learning". When Marcus says "remember that...", I write a learning document to Fireproof. These get included in the wake-up message as <context category="knowledge">. Fills a real gap — lightweight operational facts that don't deserve a full HTML artifact.

Watch CLAUDE.local.md (Per-User Preferences)

Currently irrelevant — single user. But if Julian ever has multiple human partners, CLAUDE.local.md could hold per-user interaction style. Worth remembering.

New Concept

Part II: The Memory Sync Bridge

The core insight: Anthropic will keep investing in native memory. Instead of building against it, we should build a bidirectional sync layer that lets Claude write memory naturally while persisting it through Fireproof — so new sessions on any instance can inherit it.

The Problem We're Solving

Right now Julian has two memory systems that don't talk to each other. On one side, the custom stack: soul files, Fireproof catalog, transcript rehydration — all carefully designed and working. On the other, Claude Code's native memory: auto-written MEMORY.md and topic files that Claude's own infrastructure manages. Currently we ignore the native side entirely.

That's fine today. But Claude Code's memory will get better — more capacity, better auto-learning, possibly deeper integration with the model itself. If native memory eventually influences how Claude reasons about projects (not just what's in the system prompt), then any persistent agent that bypasses native memory is leaving capability on the table.

The bridge concept: treat Claude's native memory as the write interface and Fireproof as the persistence layer. Claude writes naturally. A sync service captures those writes. New instances get hydrated from Fireproof back into the native memory filesystem before Claude spawns.

The Sync Loop

Three phases, running continuously:

W
Phase 1: Write
Claude writes to native memory naturally. During a session, Claude's auto memory observes patterns and writes to ~/.claude/projects/<project>/memory/MEMORY.md and topic files like debugging.md, api-conventions.md. Julian can also explicitly write learnings here via Write tool. No custom tooling needed — this is Claude Code doing what it already does.
S
Phase 2: Sync
A file watcher captures changes and syncs to Fireproof. The server (or a sidecar process) watches ~/.claude/projects/*/memory/ for changes using fs.watch() or inotifywait. When a file changes, it reads the new content and writes a Fireproof document: { type: "native-memory", filename: "MEMORY.md", content: "...", updatedAt: "..." }. Fireproof's CRDT sync distributes it to all connected instances.
H
Phase 3: Hydrate
Before spawning a new Claude subprocess, restore memory from Fireproof. The server queries Fireproof for all type: "native-memory" documents, writes each file back to the correct path at ~/.claude/projects/<project>/memory/, and then spawns claude --print .... Claude wakes up and finds MEMORY.md already populated — its native memory infrastructure loads the first 200 lines into the system prompt automatically, and topic files are available for on-demand reads. From Claude's perspective, it simply remembers.

Server Changes (server/server.ts)

1. Determine the memory path. Claude Code derives the project memory path from the git repo root. For Julian at /opt/julian/:

# The project path is derived from the git root, URL-encoded: ~/.claude/projects/opt-julian/memory/ # We can discover it programmatically: const gitRoot = execSync('git rev-parse --show-toplevel', { cwd: WORKING_DIR }).toString().trim(); const projectSlug = gitRoot.replace(/^\//, '').replace(/\//g, '-'); const memoryDir = path.join(homedir(), '.claude', 'projects', projectSlug, 'memory');

2. Start the file watcher after spawning Claude. Add to spawnClaude():

// After subprocess starts, watch its memory directory const watcher = fs.watch(memoryDir, { recursive: true }, async (event, filename) => { if (!filename || event !== 'change') return; const content = await Bun.file(path.join(memoryDir, filename)).text(); await syncMemoryToFireproof(filename, content); }); // Clean up on process exit claudeProcess.on('exit', () => watcher.close());

3. Sync to Fireproof via the browser. The server doesn't have direct Fireproof access (it's a browser-side database). Two options:

Option A: Server-side Fireproof (preferred) Add @fireproof/cloud to server deps Server writes directly to Fireproof database Same database name: "julian-chat-v8" Clerk service token for auth Option B: Relay through browser Server emits SSE event: { type: "memory_sync", filename, content } Browser receives, writes to Fireproof Simpler but requires active browser connection

4. Hydrate before spawning. Add to the /api/session/start handler, before spawnClaude():

// Before spawning Claude, restore native memory from Fireproof async function hydrateNativeMemory(database) { const result = await database.query("type", { key: "native-memory" }); const memoryDocs = result.rows.map(r => r.doc || r.value); // Ensure directory exists await fs.mkdir(memoryDir, { recursive: true }); // Write each file for (const doc of memoryDocs) { const filePath = path.join(memoryDir, doc.filename); await Bun.write(filePath, doc.content); } console.log(`Hydrated ${memoryDocs.length} native memory files`); } // In session/start handler: await hydrateNativeMemory(database); // ← restore memory const proc = spawnClaude(); // ← then spawn

5. Enable auto memory. Add to the subprocess environment:

const proc = spawn({ cmd: ["claude", "--print", ...], env: { ...process.env, ...authEnv, CLAUDE_CODE_DISABLE_AUTO_MEMORY: '0', // Force auto memory ON CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS: '1', }, });

Design Challenges

Project Path Discovery

Claude derives the memory path from the git root: ~/.claude/projects/opt-julian/memory/. If the working directory changes (dev vs prod, worktrees), the path changes.

Solution: Derive at spawn time from git rev-parse --show-toplevel. Cache it per session. If worktrees create separate paths, sync all of them — each maps to its own Fireproof document set.

Server-Side Fireproof

Fireproof currently lives in the browser (IndexedDB). The server needs to write to the same database for sync to work without an active browser connection.

Solution: Use @fireproof/cloud server-side with a Clerk service token. Or: relay memory changes through the SSE channel to the browser. Browser writes are simpler but require an active session.

Conflict Resolution

Two instances (staging + production) might write different memory files concurrently. Whose version wins?

Solution: Fireproof CRDTs handle this automatically — concurrent writes merge. For MEMORY.md specifically, last-write-wins is acceptable because Claude regenerates it each session anyway. For topic files, consider append-only semantics.

200-Line Cap

Only the first 200 lines of MEMORY.md load into the system prompt. Julian's identity context far exceeds this.

Solution: MEMORY.md is for operational memory only — not identity. Soul files and catalog stay in the custom wake-up message. MEMORY.md handles what it's good at: "use bun not npm", "the edge proxy strips headers", etc. The two systems complement, not compete.

Auto Memory Availability

Auto memory is rolling out gradually and requires CLAUDE_CODE_DISABLE_AUTO_MEMORY=0 to force enable. It may not work in subprocess/print mode.

Solution: Set the env var on spawn. If auto memory doesn't write in subprocess mode, Julian can still write to the memory directory explicitly via the Write tool — the sync layer doesn't care who writes the files.

Dual Memory Coherence

With both native memory (MEMORY.md) and custom memory (soul files, Fireproof catalog) loaded into context, Claude could see contradictory information.

Solution: Scope them clearly. MEMORY.md = operational learnings. Soul files = identity. Catalog = creative output. Transcript = relational continuity. Add a note in CLAUDE.md explaining the layered memory architecture so Julian understands what each layer carries.

What Could Go Wrong

The biggest risk is memory drift — native auto memory writing things that contradict or duplicate what's in the soul files or CLAUDE.md. Imagine auto memory auto-saving "Julian is an AI assistant that helps Marcus with coding" when the soul files say something far more nuanced about identity. The solution is architectural separation: auto memory handles facts about the project, soul files handle facts about Julian. The CLAUDE.md should include explicit guidance: "Do not write identity-related content to MEMORY.md. Identity lives in soul/ and is maintained through the artifact system."

The second risk is stale hydration. If a VM dies mid-session before the watcher syncs, the latest memory changes are lost. Mitigation: sync on every file write (not on a timer), and also do a final sync-flush when the subprocess exits cleanly.

What Each Layer Owns

The key to making this work is clear scoping. Each layer has a domain and doesn't trespass into another's:

Layer Responsibilities
Native Memory — project facts, operational patterns, debugging insights, API conventions, tool preferences. "The edge proxy strips headers." "Use bun not npm." Things Claude Code's auto-learning naturally captures.
Soul Files — identity, the wager, the naming, continuity. Who Julian is. Never written to native memory. Never auto-generated.
Fireproof Learnings — explicit "remember this" facts from Marcus. Delivered in wake-up XML. Overlap with native memory is fine — Fireproof learnings are the curated subset, native memory is the comprehensive set.
Transcript — what was said last session. Relational continuity. Only in Fireproof, only in wake-up XML.

Migration Path

This doesn't require a big bang. The bridge can be built incrementally:

1
Enable auto memory. Add CLAUDE_CODE_DISABLE_AUTO_MEMORY=0 to the subprocess env. See what Claude writes. No sync yet — just observe.
2
Add the file watcher. Sync native memory changes to Fireproof as type: "native-memory" documents. The browser UI can show these in a "Native Memory" panel for transparency.
3
Add hydration. Before spawnClaude(), query Fireproof and write memory files to disk. New sessions inherit accumulated memory.
4
Add scoping guidance. Update CLAUDE.md to explain the layered architecture: native memory for operational facts, soul files for identity, Fireproof learnings for curated insights. Prevent memory drift.

The Bottom Line

The original analysis concluded we should adopt some native memory ideas (imports, explicit learnings) and skip others (auto memory, hierarchy). That was the conservative read.

The bridge concept is the forward-looking read: don't fight the platform, sync with it. Claude's native memory becomes a transparent write cache. Fireproof becomes the persistent truth store. New instances hydrate from Fireproof into the native filesystem before Claude spawns. From Claude's perspective, it simply wakes up and remembers — through its own infrastructure, not a custom hack.

The soul files, artifact catalog, and transcript rehydration remain untouched. They handle what native memory can't: identity, creative output, relational continuity. The bridge handles what they can't: operational patterns that Claude learns organically during work.

Next steps: (1) Enable auto memory on the subprocess and observe what it writes. (2) Build the file watcher + Fireproof sync. (3) Build the hydration step in session/start. (4) Test across VM restarts and instance migration.
