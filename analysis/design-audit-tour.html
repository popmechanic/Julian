Wave 01

Transition & Animation Cleanup

This was the single highest-impact change. Before the audit, Julian's transitions were a patchwork — mixed durations (0.1s here, 300ms there, 0.2s somewhere else), generic easing (plain ease everywhere), and the cardinal sin of frontend animation: transition: all used in 14 places across the three files.

transition: all is a foot-gun. It means "animate every CSS property that changes," which causes unexpected flickers when unrelated properties change and forces the browser to check every property on every frame. Worse, it makes the UI feel mushy — everything moves at the same speed in the same way, so nothing has visual hierarchy.

The fix was systematic. First, I added CSS custom properties to the root stylesheet for shared timing values: a natural --ease-out curve, a snappy --ease-out-quart, and three duration tiers (100ms for micro-interactions, 150ms for standard UI, 250ms for larger state changes). Then I went through every single transition: all and replaced it with the specific properties that actually change on that element.

I also expanded the reduced-motion media query from a selective list of four animations to a universal blanket rule that disables all animations and transitions site-wide for users who prefer reduced motion. And in vibes.jsx, the shredCard and collapseToLine keyframes — the card destruction animation — now have their own reduced-motion override.

Wave 02

Button & Interactive Feel

Buttons are how you talk to Julian. They should feel responsive — press down on click, show focus when navigated by keyboard, tell you when they're disabled. Before the audit, buttons had inconsistent feedback: some had no press effect, focus was invisible on keyboard navigation, disabled states used cursor: default (which just looks like nothing happened), and touch targets on mobile were too small to hit reliably.

Every button now gets a global scale(0.97) on press via CSS :active. Keyboard focus shows a gold outline with 2px offset — visible but not intrusive. All disabled buttons across the app now use cursor: not-allowed instead of cursor: default, which gives the user clear visual feedback that the button exists but isn't available right now.

The send button got an aria-label="Send message" since it only displays the letter "A" — screen readers had no idea what it did. The setup screen tabs got proper role="tab" and aria-selected attributes. Small things, but they make the app navigable without a mouse.

chat.jsx
ARIA labels, tab roles, disabled cursors (7 buttons fixed)
Send button: aria-label="Send message". Setup tabs: role="tablist" container, role="tab" + aria-selected on each tab. Seven disabled buttons changed from cursor: 'default' to cursor: 'not-allowed'.
Wave 03

Smooth State Transitions

Julian's biggest "dead" moment was the transition between app states. You'd see the yellow boot screen with "BOOTING...", and then in a single frame — snap — the dark chat UI. No crossfade, no easing, just an instant swap. It felt like a channel change, not a living presence waking up.

The fix is a CSS class called .app-state-enter that applies a simple opacity fade using the shared --duration-slow (250ms) and --ease-out curve. Each of the three app states — loading, mobile layout, desktop layout — gets this class with a unique key prop so React triggers a remount animation when switching between them.

In the chat itself, tool call blocks now fade in with the same message-enter animation that regular messages use, instead of appearing abruptly. And the streaming cursor (the blinking underscore at the end of Julian's messages) now always exists in the DOM with visibility toggling instead of conditional rendering. This prevents a tiny layout shift every time the cursor appears or disappears.

Both message containers (mobile and desktop) got scrollBehavior: 'smooth', so auto-scrolling to new messages feels fluid instead of jumping.

Crossfade between boot/setup/chat states, smooth scroll
New @keyframes fadeIn + .app-state-enter class. Applied to all three conditional branches with unique key attributes. scrollBehavior: 'smooth' added to both mobile and desktop message containers.
chat.jsx

Wave 04

Status Indicators Come Alive

The StatusDots component is Julian's heartbeat — three small squares in the header that tell you if he's connected and listening. Before the audit, when Julian was offline, the dots just turned #333 — effectively invisible against the dark background. The user had to read the "OFFLINE" text to know something was wrong.

Now when disconnected, all three dots glow red with a staggered pulse animation. The first dot pulses at full intensity, the second at 50% opacity with a 200ms delay, the third at 30% with a 400ms delay. It creates a wave effect that catches your eye without being alarming. The JulianScreen connection dot got the same treatment — red with pulse when the WebSocket is down.

The status text ("PROCESSING...", "LISTENING", "OFFLINE") had its opacity bumped from 0.6 to 0.8. At 0.6 against the dark background, the #AA8800 color was washing out to near-invisible. 0.8 keeps the visual hierarchy (it's still secondary to the "JULIAN" title) while being actually readable.

Wave 05

Touch & Mobile Polish

Julian should feel as good on a phone as on a desktop. The main issues were: hover styles sticking on touch (you'd tap a grid item and the hover effect would stay on until you tapped something else), tap targets too small (the header buttons were tiny 3px-padded elements), and the chat input running spellcheck on what's essentially a terminal.

The hover fix was structural. The original CSS defined hover effects unconditionally and then tried to undo them with @media (hover: none). This is backwards — it means touch devices still get the hover flash on tap. I flipped it: hover effects now only apply inside @media (hover: hover) and (pointer: fine), meaning they only activate on devices that actually support hover (mice, trackpads). Touch devices never see them.

The chat textarea got spellCheck={false} and autoComplete="off". Julian's input is styled as a retro terminal in all-caps — red squiggly underlines from the browser spellchecker break the illusion completely.

index.html
Hover effects gated behind @media (hover: hover)
Removed the unconditional .screen-grid-item:hover rules. Moved them inside @media (hover: hover) and (pointer: fine). Removed the old @media (hover: none) override block — no longer needed.
chat.jsx
Disabled spellcheck on chat input, 44px touch targets
ChatInput textarea: spellCheck={false}, autoComplete="off". Mobile header buttons already got 44px in Wave 2.
Wave 06

Visual Details & Delight

The final wave is small polish that makes Julian feel crafted rather than generated. None of these are critical in isolation, but together they create the feeling that someone cared about the details.

Text selection now matches the gold theme — select any text and it highlights in translucent gold instead of the browser's default blue. The decorative X characters in the corners of the page got user-select: none so you can't accidentally select them. Placeholder text contrast went from 0.4 to 0.55 alpha — enough to read but still clearly placeholder.

The ThinkingDots component got a complete rework. Before, it was just "PROCESSING" with a blinking underscore — the same cursor used everywhere else. Now it's three small gold squares that pulse in sequence with staggered timing, creating a wave effect that reads as "working on it" rather than just "blinking."

ToolCallBlock borders went from 2px #AA8800 to 3px #C8A800 — brighter and thicker, so tool calls are visually distinct from regular text instead of fading into the background.
